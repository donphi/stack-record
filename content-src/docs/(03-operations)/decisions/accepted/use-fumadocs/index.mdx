## Decision

Adopt Fumadocs as the publishing layer for the Stack Record knowledge wiki instead of building or adopting any other documentation framework.

## Context

The [[ukb_pipeline]] project generates a growing body of research notes, architecture decisions, and experimental records. These notes live in Obsidian as Markdown with YAML frontmatter and wiki-link cross-references. A publishing layer is needed so the knowledge base can be browsed, searched, and shared outside of Obsidian without losing structural metadata or math rendering.

Key requirements:

- **MDX support** — notes use JSX components for callouts, tables, and interactive diagrams.
- **Sidecar metadata** — each note carries a `meta.json` with fields like `closure_score`, `review_cycle`, and `related_projects`. The framework must consume these without requiring frontmatter duplication.
- **Math rendering** — KaTeX or MathJax for inline and block equations is non-negotiable.
- **React component embedding** — custom components such as the library registry table and knowledge-graph viewer must render inside pages.
- **Search** — full-text search across several hundred pages with acceptable cold-start latency.

## Why this decision exists

Without a publishing layer, the wiki is locked inside a single-user Obsidian vault. Collaborators, reviewers, and future-self need a web-accessible view that preserves links, metadata, and formatting. Choosing the wrong framework early creates expensive migration debt once the vault grows past a few hundred notes.

## Options considered

### Option A — Custom Next.js SSG

Build a bespoke static site generator on top of Next.js App Router using `next-mdx-remote` and a hand-rolled content layer.

**Pros:**
- Total control over routing, metadata handling, and rendering pipeline.
- No upstream dependency risk.

**Cons:**
- Requires implementing breadcrumbs, sidebar generation, search index, and table-of-contents from scratch.
- Estimated 3–4 weeks of initial build time plus ongoing maintenance.
- Every new MDX plugin or remark/rehype integration must be wired up manually.

### Option B — Docusaurus

Use Meta's Docusaurus framework with its built-in docs plugin.

**Pros:**
- Large community and plugin ecosystem.
- Built-in versioning, search (Algolia), and i18n.

**Cons:**
- Docs plugin expects a rigid sidebar configuration format that conflicts with the sidecar `meta.json` pattern.
- React component customisation requires swizzling, which breaks on major upgrades.
- No native support for consuming arbitrary JSON metadata per page.

### Option C — MkDocs Material

Python-based static site generator with the Material theme.

**Pros:**
- Excellent search, admonitions, and navigation out of the box.
- Well-suited for pure Markdown documentation.

**Cons:**
- No native MDX support — JSX components would need to be replaced with Jinja macros or iframes.
- Python toolchain adds a second language runtime to the build pipeline.
- Sidecar JSON consumption requires a custom plugin with no existing reference implementation.

### Option D — Obsidian Publish

Use Obsidian's first-party publishing service.

**Pros:**
- Zero build step; publishes directly from the vault.
- Preserves wiki-links and graph view natively.

**Cons:**
- Closed-source hosting with no self-hosting option.
- No custom React components.
- Limited control over page layout, metadata display, and search behaviour.
- Ongoing subscription cost with no data portability guarantee.

## Why the chosen option won

Fumadocs won because it satisfies every requirement with minimal custom code:

1. **Native MDX** — Fumadocs is built on Next.js App Router and treats `.mdx` as first-class content. Remark and rehype plugins slot in via `next.config.mjs` with no wrapper library.
2. **Sidecar `meta.json`** — Fumadocs already uses `meta.json` files for sidebar ordering and page grouping. Extending the schema with custom fields like `closure_score` and `review_cycle` requires only a type-augmentation file, not a fork.
3. **Plugin system** — `fumadocs-core` exposes a source API that can consume any content directory structure. The materializer script transforms Obsidian notes into the expected layout without touching Fumadocs internals.
4. **React components** — since Fumadocs pages are React Server Components, embedding custom components (library table, graph viewer) is straightforward.
5. **Search** — Fumadocs ships with a built-in search API backed by Orama, which indexes content at build time and serves queries client-side with sub-50ms latency.
6. **Active maintenance** — the project is actively maintained with frequent releases and a responsive maintainer.

## Tradeoffs accepted

- **Tied to Next.js** — if Next.js introduces breaking changes in App Router, the wiki build may require coordinated upgrades.
- **Smaller community** — Fumadocs has a fraction of Docusaurus's community, so troubleshooting edge cases relies more on reading source code than on Stack Overflow answers.
- **Sidecar materializer** — the Obsidian-to-Fumadocs pipeline requires a custom script (`materialize.ts`) that resolves wiki-links, strips frontmatter, and emits `meta.json`. This is net-new code that must be maintained.
- **Docker deployment** — the site is deployed as a Docker image rather than static HTML, which increases infrastructure complexity slightly compared to a pure SSG.

## Consequences

1. A materializer script converts Obsidian vault content into the Fumadocs-compatible `content-src/` directory structure.
2. Each note gets an `index.mdx` (body only, no frontmatter) and an `index.meta.json` (all metadata).
3. The Docker image runs `next build && next start` behind a reverse proxy.
4. Custom components are stored in `components/` and imported via MDX shortcodes.
5. Search indexing happens at build time — no external search service is needed.
6. Future notes added in Obsidian are automatically published on the next build cycle.

## Where this has been used

- [[ukb_pipeline]] — the primary project whose documentation motivated this decision.
- The Stack Record wiki itself is the running implementation of this ADR.

## Review trigger

Re-evaluate if any of the following occur:

- Fumadocs releases a major version that removes or restructures the `meta.json` sidecar pattern.
- The wiki grows beyond 1,000 pages and build times exceed 5 minutes.
- Next.js App Router introduces breaking changes that require more than a day of migration effort.
- A competing framework gains native sidecar-metadata support with a larger community.
