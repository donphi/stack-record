# Authoring Templates

Every note in Stack Record is a pair: a **body-only `.mdx`** file and a **`.meta.json` sidecar**. The materialiser joins them at build time. No `.mdx` file under `content-src/` may contain frontmatter.

<Callout type="warn">
Every `.mdx` must have a matching `.meta.json`. Every `.meta.json` must have `title`, `description`, `id`, and `type`. The materialiser will reject violations.
</Callout>

## Writing rules

<Steps>
<Step>
### Define before explaining
State what the thing is before explaining why or how.
</Step>
<Step>
### Use stable section order
Follow the template for the note type exactly. Omit inapplicable sections but preserve order.
</Step>
<Step>
### Plain-English interpretation
Every technical note must include a plain-language summary.
</Step>
<Step>
### One worked example minimum
Show at least one concrete example with real values.
</Step>
<Step>
### Show drawbacks
No note should only sell the idea. State limitations and failure modes.
</Step>
<Step>
### Link outward
Connect to related notes, methods, systems, and projects.
</Step>
<Step>
### Write for decision utility
The reader should leave knowing what to do.
</Step>
</Steps>

<Tabs items={["Body Template", "Sidecar Template", "Closure Checklist"]}>
<Tab value="Body Template">

```md title="concept-name.mdx"
## What this is
## Why this matters
## Where this fits
```

</Tab>
<Tab value="Closure Checklist">

<Accordions>
<Accordion title="Concept closure requirements">

- One parent map
- Two related or prerequisite concepts
- One child, example, or comparison note
- One related method
- One related system or project link
- Drawbacks or failure modes stated
- Open questions listed
- Review metadata present

</Accordion>

## Reference

<Tabs items={["Body Template", "Sidecar Template", "Closure Checklist"]}>
<Tab value="Body Template">

Reference pages have no fixed section order. Use whatever structure best serves the content (glossary, notation tables, template listings, etc.). The following is a suggested starting point:

```md title="reference-name.mdx"
## Overview
## Entries
## Usage notes
## Related notes
```

<Callout type="info">
Reference pages are flexible. Use tables, definition lists, or whatever format best suits the reference material.
</Callout>

</Tab>
<Tab value="Sidecar Template">

```json title="reference-name.meta.json"

  "description": "",
  "id": "REF-{{CODE}}",
  "type": "reference",
  "domain": "",
  "status": "evergreen",
  "tags": ["reference/{{TAG}}"],
```

## Fumadocs MDX tips

<Callout type="info">

**Available components** (no imports needed in `.mdx` files):

- `<Steps>` / `<Step>` — numbered step guides
- `<Tabs>` / `<Tab>` — tabbed content switching
- `<Callout>` — info, warn, error, success boxes
- `<Accordion>` / `<Accordions>` — expandable sections
- `<Cards>` / `<Card>` — link card grids
- `<Files>` / `<File>` / `<Folder>` — file tree diagrams
- `<TypeTable>` — prop/field documentation tables
- `<ImageZoom>` — zoomable images

**Code block features** (no components needed):

- ` ```ts title="filename.ts" ` — titled code blocks
- ` ```ts lineNumbers ` — line numbers
- `// [!code highlight]` — line highlighting
- `// [!code --]` / `// [!code ++]` — diff markers
- `// [!code focus]` — focus lines
- `$$...$$` — KaTeX math blocks

</Callout>
