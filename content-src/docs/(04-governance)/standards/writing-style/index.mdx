## Purpose

This standard defines how every note in Stack Record is written, including the seven mandatory writing rules, the MDX artifacts available for making content engaging and scannable, and where each artifact is practical across the nine note types.

## Applies to

All notes in all sections: concepts, methods, systems, decisions, experiments, projects, maps, standards, and references.

## The seven writing rules

<Steps>
<Step>

### Define before explaining

State what the thing is in the first sentence. Do not build up to a definition. The reader should know what they are looking at within five seconds.

</Step>
<Step>

### Use stable section order

Every note type has a fixed template. Never reorder sections. A reader who has seen one concept note knows where to find the worked example in every other concept note.

</Step>
<Step>

### Use plain-English interpretation

After every equation, algorithm, or formal rule, write one sentence that says what it does in plain English. If you cannot explain it simply, you do not understand it well enough to record it.

</Step>
<Step>

### Show one worked example

Every concept, method, and system note must include at least one concrete example with real values. Abstract descriptions without examples are not complete.

</Step>
<Step>

### Show drawbacks

No note only sells the idea. Every concept has failure modes. Every method has limitations. Every system has tradeoffs. If the drawbacks section is empty, the note is not honest.

</Step>
<Step>

### Link outward

Every note must connect to at least two other notes. Isolated notes are orphans. Use `[[wiki_link]]` syntax for cross-references — the materialiser resolves them to real links at build time.

</Step>
<Step>

### Write for decision utility

The reader should leave every note knowing what to do. Not just what something is, but when to use it, when not to, and what to try next.

</Step>
</Steps>

## Type-specific additions

<Tabs items={['Concept / Math', 'Method', 'System', 'Decision', 'Experiment']}>
<Tab value="Concept / Math">

Math and concept notes additionally require:

- Equation in `$$...$$` KaTeX block
- Breakdown section with variable definitions
- Plain-English interpretation of the equation
- Worked example with real numbers
- ASCII diagram where spatial relationships matter

</Tab>
<Tab value="Method">

Method notes additionally require:

- Inputs and outputs sections (what goes in, what comes out)
- Procedure as numbered steps
- Code sketch with a titled code block
- Validation and evidence section

</Tab>
<Tab value="System">

System notes additionally require:

- Architecture diagram (ASCII or `<Files>` tree)
- Configuration contract (what must match what)
- Failure modes section
- Monitoring section (what to watch)

</Tab>
<Tab value="Decision">

Decision notes additionally require:

- Options considered (at least two)
- Why the chosen option won
- Tradeoffs accepted (what you gave up)
- Review trigger (when to revisit)

</Tab>
<Tab value="Experiment">

Experiment notes additionally require:

- Hypothesis stated before results
- Setup and procedure separated
- Threats to validity
- What this changes (feedback to other notes)

</Tab>
</Tabs>

### Tabs

Switchable panels for showing alternatives, comparisons, or different views of the same content.

<Tabs items={['Cosine Similarity', 'Dot Product', 'BM25']}>
<Tab value="Cosine Similarity">

Measures the angle between two vectors. Ignores magnitude — only direction matters.

$$
\cos(\theta) = \frac{\mathbf{a}\cdot\mathbf{b}}{\|\mathbf{a}\|\|\mathbf{b}\|}
$$

Best when: vectors are L2-normalised and you care about semantic direction, not confidence.

</Tab>
<Tab value="Dot Product">

Combines direction and magnitude into a single score. Larger vectors score higher.

$$
\mathbf{a}\cdot\mathbf{b} = \sum_{i=1}^{n} a_i b_i
$$

Best when: vector norm carries meaningful signal (e.g. confidence or frequency).

</Tab>
<Tab value="BM25">

A corpus-aware scoring function that uses term frequency, inverse document frequency, and document length normalisation.

$$
\text{BM25}(q, d) = \sum_{t \in q} \text{IDF}(t) \cdot \frac{f(t,d) \cdot (k_1 + 1)}{f(t,d) + k_1 \cdot (1 - b + b \cdot \frac{|d|}{\text{avgdl}})}
$$

Best when: you need corpus-level rarity signals that pretrained embeddings cannot provide.

</Tab>
</Tabs>

### Code blocks

Built into the MDX pipeline. Supports titles, line numbers, highlighting, diffs, and focus.

```python title="rrf.py" lineNumbers
def rrf(rank_lists, k=60):
    scores = {}
    for ranking in rank_lists:
        for idx, item in enumerate(ranking, start=1):
            scores[item] = scores.get(item, 0.0) + 1.0 / (k + idx)  # [!code highlight]
    return dict(sorted(scores.items(), key=lambda x: x[1], reverse=True))
```

Line diff annotations show additions and removals:

```python title="config_loader.py"
def validate(config, stage):
    required = GLOBAL_REQUIRED_FIELDS  # [!code --]
    required = STAGE_REQUIRED_FIELDS[stage]  # [!code ++]
    for field in required:
        if field not in config:
            raise ValueError(f"Missing {field}")
```

Available annotations (place at end of line inside the code fence):

| Annotation | Effect |
|---|---|
| `// [!code highlight]` | Highlights the line |
| `// [!code ++]` | Shows the line as an addition (green) |
| `// [!code --]` | Shows the line as a removal (red) |
| `// [!code focus]` | Dims all other lines, focuses this one |
| `// [!code word:term]` | Highlights every occurrence of `term` |

## KaTeX math

Provided by `remark-math` + `rehype-katex`. Use `$...$` for inline and `$$...$$` for display.

Inline example: The RRF constant is typically $k = 60$, giving each rank position a contribution of $\frac{1}{60 + r}$.

Display example:

$$
RRF(d) = \sum_{i=1}^{n} \frac{1}{k + rank_i(d)}
$$

$$
\cos(\theta) = \frac{\mathbf{a}\cdot\mathbf{b}}{\|\mathbf{a}\|\|\mathbf{b}\|}
$$

## Failure conditions

<Callout title="Build failure" type="error">
The materialiser will reject any `.mdx` file under `content-src/` that begins with `---` (frontmatter). All metadata belongs in the `.meta.json` sidecar.
</Callout>

<Callout title="Silent rendering failure" type="warn">
If a component tag is misspelled or not registered in `components/mdx.tsx`, it will render as plain text or cause a build error. Always use the exact tag names listed above.
</Callout>

- A note missing any of the seven writing rules is incomplete
- A concept note without a worked example cannot reach closure
- A method note without a procedure section cannot reach closure
- A system note without failure modes cannot reach closure
- Inline `style={}` props with hardcoded colours are forbidden — all styling comes from `app/global.css`
