## Question

Does adding a second independent retrieval judge and fusing their rankings via [[rrf_pipeline]] improve Precision@10 compared to a single-judge baseline on the [[ukb_pipeline]] biomedical corpus?

## Hypothesis

Fusing rankings from two retrieval judges — one dense (embedding-based) and one sparse (BM25) — using Reciprocal Rank Fusion will improve Precision@10 by at least 10 percentage points over the single dense-retrieval baseline, because the two judges make complementary errors: dense retrieval misses exact-match keyword queries while BM25 misses semantic paraphrases.

## Why this matters

The [[ukb_pipeline]] retrieval stage feeds candidates into a downstream extraction pipeline. False positives waste compute on irrelevant documents; false negatives cause information loss that cannot be recovered later. A single retrieval judge (dense cosine similarity via [[cosine_similarity]]) achieved 0.72 Precision@10 on the evaluation set, which left meaningful room for improvement. If a two-judge approach closes that gap without doubling latency, the pipeline benefits at acceptable cost.

## Setup

### Judges

| Judge | Type | Implementation |
|---|---|---|
| Judge A (dense) | Embedding cosine similarity | PubMedBERT-base (768-dim) with FAISS IVF index |
| Judge B (sparse) | BM25 | Elasticsearch 8.x with biomedical stopword list |

### Fusion method

Reciprocal Rank Fusion (RRF) as described in the [[rrf_pipeline]] method note. The fused score for document $d$ is:

$$
\text{RRF}(d) = \sum_{j \in \{A,B\}} \frac{1}{k + \text{rank}_j(d)}
$$

where $k = 60$ (standard RRF constant).

### Corpus and query set

- Corpus: 50,000 PubMed abstracts from the UK Biobank phenotype-linked subset.
- Query set: 200 curated queries with binary relevance labels (same set used across retrieval experiments).

### Infrastructure

- Dense retrieval: FAISS on A100 GPU via [[gpu_worker_pool]].
- Sparse retrieval: Elasticsearch running on a 4-core VM with 16 GB RAM.
- Fusion and evaluation: Python script orchestrated via [[rabbitmq_central_broker]] task queue.

## Procedure

1. Run all 200 queries through Judge A (dense). Retrieve top-50 candidates per query.
2. Run all 200 queries through Judge B (BM25). Retrieve top-50 candidates per query.
3. Fuse the two ranked lists per query using RRF ($k = 60$).
4. Evaluate fused ranking: Precision@10, Recall@50, and MRR.
5. Compare against single-judge baselines (Judge A alone, Judge B alone).
6. Run paired bootstrap significance test (10,000 iterations) on Precision@10 differences.
7. Measure end-to-end latency: single-judge vs. two-judge pipeline.

## Results

| Metric | Judge A (dense) | Judge B (BM25) | RRF fusion |
|---|---|---|---|
| Precision@10 | 0.72 | 0.68 | **0.89** |
| Recall@50 | 0.81 | 0.76 | **0.93** |
| MRR | 0.65 | 0.61 | **0.82** |
| p99 latency (ms) | 42 | 28 | 58 |

The Precision@10 improvement from 0.72 (dense baseline) to 0.89 (RRF fusion) is statistically significant ($p < 0.001$, paired bootstrap, 10,000 iterations). The 17-point gain exceeds the 10-point hypothesis threshold.

End-to-end latency increased from 42ms to 58ms — a 38% increase — because the two judges run in parallel and fusion adds minimal overhead.

## Interpretation

The two-judge approach works because the judges fail on different query types:

- **Dense retrieval** excels at semantic paraphrases ("heart attack" retrieves "myocardial infarction") but struggles with queries containing rare acronyms or exact-match identifiers (gene symbols, drug codes).
- **BM25** handles exact-match terms reliably but misses semantically equivalent phrases that use different vocabulary.

RRF combines these complementary strengths. Documents ranked highly by both judges receive a strong fused score, while documents ranked highly by only one judge still appear in the top results if the other judge does not actively rank them low.

The 38% latency increase is acceptable because the two judges run in parallel via the [[rabbitmq_central_broker]] task queue, and the absolute latency (58ms p99) remains well within the pipeline's 200ms budget per query.

## Threats to validity

- **Query set size** — 200 queries may not represent the full distribution of production queries. Performance on rare biomedical subdomains (e.g., pharmacogenomics) is undersampled.
- **BM25 tuning** — the biomedical stopword list and Elasticsearch analyser were tuned on a development split. Different tuning choices could shift Judge B's contribution.
- **RRF constant sensitivity** — $k = 60$ is the standard value but was not optimised for this corpus. A grid search over $k$ might yield further gains or reveal sensitivity.
- **Relevance label quality** — binary labels were assigned by a single domain expert. Inter-annotator agreement was not measured.

## What this changes

1. The [[ukb_pipeline]] retrieval stage was updated to use two-judge RRF fusion as the default retrieval method.
2. The [[rrf_pipeline]] method note was promoted from draft to evergreen based on this empirical validation.
3. Elasticsearch was added as a required service in the pipeline's Docker Compose configuration.
4. The retrieval latency budget was revised from 50ms to 75ms to accommodate the two-judge architecture with headroom.

## Final status

Completed — hypothesis confirmed. Two-judge RRF fusion is now the production retrieval strategy for [[ukb_pipeline]].
