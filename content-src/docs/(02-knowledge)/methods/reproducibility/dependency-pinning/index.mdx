## What this method is

Dependency pinning is the practice of locking every software dependency to an exact version (or content hash) so that builds are deterministic and reproducible across machines and time. Instead of declaring loose version ranges like `requests>=2.28`, a pinned specification reads `requests==2.31.0` with an accompanying integrity hash. Lockfiles (`poetry.lock`, `pnpm-lock.yaml`, `Pipfile.lock`) encode the fully resolved dependency tree, including transitive dependencies, so that every install reproduces the same environment byte-for-byte.

## Why this is used

Without pinning, a `pip install` or `npm install` run today and tomorrow can produce different environments because upstream packages release new versions continuously. This "works on my machine" drift is a major source of irreproducible results in computational research and silent regressions in production services.

In research pipelines, reproducibility is a first-class requirement. If a model evaluation depends on a specific version of NumPy, a silent upgrade that changes floating-point behaviour can alter results without any code change. Dependency pinning eliminates this class of failure.

## Where this fits

Dependency pinning operates at the build and environment layer of a software system:

1. Source code → **Dependency resolution + pinning** → Lockfile
2. Lockfile → Deterministic install → Identical environment
3. Identical environment → [[docker_multi_stage]] build → Reproducible container image

It is a prerequisite for reproducible [[docker_multi_stage]] builds and is used across all projects in the Stack Record ecosystem, including the [[ukb_pipeline]].

## Inputs

| Input | Description |
|---|---|
| Dependency manifest | A file declaring direct dependencies and their version constraints (`pyproject.toml`, `package.json`, `requirements.in`). |
| Package registry | The upstream index (PyPI, npm, crates.io) from which packages are resolved. |
| Resolver | A dependency resolver that computes a compatible set of versions for the full transitive tree (pip-tools, Poetry, pnpm). |

## Outputs

| Output | Description |
|---|---|
| Lockfile | A machine-generated file listing every resolved package, its exact version, and integrity hashes (`poetry.lock`, `pnpm-lock.yaml`). |
| Pinned requirements | A flat list of `package==version` entries, optionally with `--hash` flags (`requirements.txt` generated by `pip-compile`). |

## Procedure

1. **Declare direct dependencies** in a manifest file with loose constraints that express compatibility intent (e.g. `numpy >=1.24,<2`).
2. **Run the resolver** to compute a full dependency tree that satisfies all constraints:
   - Python: `poetry lock` or `pip-compile requirements.in -o requirements.txt`
   - Node: `pnpm install --frozen-lockfile=false` (generates `pnpm-lock.yaml`)
3. **Commit the lockfile** to version control alongside the manifest. The lockfile is the source of truth for exact versions.
4. **Install from the lockfile** in CI and production: `poetry install --no-update`, `pip install -r requirements.txt`, or `pnpm install --frozen-lockfile`.
5. **Update deliberately** by running the resolver again when you intentionally want newer versions. Review the diff in the lockfile before committing.
6. **Verify hashes** where supported (`pip install --require-hashes -r requirements.txt`) to guard against registry tampering.

## Formal rule or formula

A pinned environment $E$ is defined as a set of packages with exact versions:

$$E = \{(p_i, v_i) \mid i = 1, \ldots, n\}$$

Reproducibility holds when:

$$\text{install}(E, t_1) = \text{install}(E, t_2) \quad \forall \; t_1, t_2$$

This equality is guaranteed if and only if the version $v_i$ uniquely identifies an immutable artifact (enforced by hash verification). If a registry allows re-uploading the same version with different contents, hash pinning is necessary to detect the discrepancy.

## Why it works

Package registries assign unique version identifiers to published artifacts and (in well-behaved registries) treat published versions as immutable. By recording exact versions, the lockfile transforms a non-deterministic resolution process into a deterministic lookup. Hash verification adds a second layer of assurance by checking the artifact's content against a known digest, defending against both registry compromise and version yanking/re-upload.

## Code sketch

```bash
# Python with pip-tools
pip install pip-tools
echo "numpy>=1.24,<2" > requirements.in
pip-compile requirements.in --generate-hashes -o requirements.txt
# requirements.txt now contains:
#   numpy==1.26.4 \
#     --hash=sha256:...

pip install --require-hashes -r requirements.txt

# Python with Poetry
poetry init
poetry add "numpy>=1.24,<2"
poetry lock          # generates poetry.lock
poetry install       # installs from lockfile

# Node with pnpm
pnpm init
pnpm add lodash
# pnpm-lock.yaml generated automatically
pnpm install --frozen-lockfile   # CI mode: fails if lockfile is out of date
```

## Benefits

- Eliminates "works on my machine" drift between development, CI, and production.
- Makes builds deterministic, which is essential for scientific reproducibility.
- Hash verification guards against supply-chain attacks and registry compromise.
- Lockfile diffs in pull requests make dependency changes explicit and reviewable.

## Drawbacks

- Lockfiles can be large and produce noisy diffs, especially in monorepos with many packages.
- Pinning delays security patches: a pinned vulnerable version stays until someone explicitly updates.
- Resolver conflicts can be frustrating when transitive dependencies have incompatible version constraints.
- Multiple pinning tools in a single project (e.g. pip freeze + Poetry) can lead to conflicting sources of truth.

## Failure modes

| Failure | Cause | Mitigation |
|---|---|---|
| Stale pins | Pinned versions accumulate known vulnerabilities over time. | Schedule automated update PRs (Dependabot, Renovate) with CI checks. |
| Lockfile not committed | Developer forgets to commit the lockfile; CI resolves from scratch. | CI step: `--frozen-lockfile` or `--require-hashes` fails on mismatch. |
| Yanked package | A maintainer removes a published version from the registry. | Mirror critical dependencies or use a private registry proxy. |
| Hash mismatch | Registry serves a different artifact for the same version. | `--require-hashes` flag rejects the install; investigate before proceeding. |
| Transitive conflict | Two direct dependencies require incompatible versions of a shared transitive. | Use a resolver with clear conflict reporting; consider forking or upgrading. |

## Where I have used it

All Python projects in the Stack Record ecosystem use `pip-compile` with `--generate-hashes` to produce pinned requirements files. The [[ukb_pipeline]] Dockerfile copies `requirements.txt` into the builder stage of a [[docker_multi_stage]] build to ensure the production image contains exactly the tested dependency set. Node-based projects (including this wiki site) use `pnpm install --frozen-lockfile` in CI.

## Validation and evidence

- The Python Packaging Authority (PyPA) recommends hash-checking mode for secure, reproducible installs.
- Reproducible Builds project (reproducible-builds.org) documents that pinning is a necessary (though not sufficient) condition for bit-for-bit reproducible builds.
- The 2024 xz-utils backdoor incident demonstrated that supply-chain integrity checks (including hash verification) are essential even for well-known packages.
