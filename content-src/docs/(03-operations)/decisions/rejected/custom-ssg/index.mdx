## Decision

Build a bespoke static site generator on Next.js App Router with hand-rolled content loading, sidebar generation, and search.

## Context

The [[ukb_pipeline]] wiki needed a publishing layer that could render MDX, consume sidecar `meta.json` files, and embed React components. A custom SSG was the first approach evaluated because it promised total control over every aspect of the build pipeline.

The prototype used `next-mdx-remote` for MDX compilation, a recursive directory walker for sidebar generation, and a custom Flexsearch index built at `next build` time. After two weeks of development, the scope of remaining work became clear enough to compare against off-the-shelf alternatives.

## Why this decision exists

A custom SSG was the default starting point because no existing framework appeared to support the sidecar `meta.json` pattern natively. This ADR records why the approach was abandoned so the reasoning is preserved if the question resurfaces.

## Options considered

### Option A — Custom Next.js SSG (this proposal)

Hand-build everything: content loader, sidebar, breadcrumbs, search, table of contents, pagination, and link resolution.

### Option B — Fumadocs

Adopt a framework that already implements content loading, sidebar generation, and search on top of Next.js App Router. See [[use_fumadocs]] for the full evaluation.

### Option C — Docusaurus

Adopt Meta's documentation framework. Strong community but rigid sidebar configuration and limited sidecar metadata support.

### Option D — MkDocs Material

Python-based generator with excellent out-of-the-box theming. No MDX support and a second language runtime in the build chain.

## Why the chosen option won

The custom SSG did not win. It was rejected in favour of Fumadocs. The full rationale for the winning option is recorded in [[use_fumadocs]].

## Tradeoffs accepted

By rejecting the custom SSG, the following tradeoffs were accepted:

- **Upstream dependency** — the wiki now depends on Fumadocs release cadence and design decisions.
- **Less control** — edge-case rendering behaviours are constrained by what Fumadocs exposes through its API.
- **Learning curve** — contributors must understand Fumadocs conventions rather than a fully bespoke codebase.

These tradeoffs were judged acceptable because they are outweighed by the development-time savings and reduced ongoing maintenance.

## Consequences

1. **The prototype was archived.** Two weeks of custom SSG code was discarded. The routing logic and Flexsearch integration were not portable.
2. **Maintenance burden eliminated.** Sidebar generation, breadcrumb rendering, table-of-contents extraction, and search indexing are now handled by Fumadocs rather than bespoke code.
3. **Reinvention avoided.** The custom SSG was on track to reimplement features that Fumadocs already provides: MDX compilation, `meta.json` sidebar ordering, and client-side search via Orama.
4. **Opportunity cost recovered.** The time saved was redirected to building the materializer pipeline and writing actual wiki content rather than framework plumbing.

## Where this has been used

Nowhere — the custom SSG was rejected before it reached production. The two-week prototype was used only for internal evaluation.

## Review trigger

Re-evaluate if Fumadocs is abandoned or introduces breaking changes severe enough that maintaining a fork exceeds the cost of building a replacement. See [[use_fumadocs]] for specific trigger conditions.
