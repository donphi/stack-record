## Purpose

This page is the complete reference for writing Stack Record notes. It is designed to be injected as context for an LLM writing assistant (via `@llm-writing-guide` in Obsidian Copilot) and also serves as on-site documentation.

It covers: body templates per note type, all sidecar connection fields, all constrained enum values, MDX artifact syntax and placement guidance, closure rules, and writing quality standards.

## B. Sidecar fields and knowledge graph connections

Every `.mdx` must have a matching `.meta.json` sidecar with the same stem (e.g. `index.mdx` ↔ `index.meta.json`). Each article lives in its own folder: `slug/index.mdx` + `slug/index.meta.json`. The materialiser will reject any mismatch.

### Required fields (build fails without these)

| Field | Type | Example |
|-------|------|---------|
| `title` | string | `"Cosine Similarity"` |
| `description` | string | `"Angle-based vector similarity measure."` |
| `id` | string | `"C-IR-0007"` |
| `type` | enum | See allowed values below |

### Constrained enum values

**`type`** (required): `map`, `concept`, `method`, `system`, `decision`, `experiment`, `project`, `standard`, `reference`

**`status`**: `draft`, `active`, `evergreen`, `enforced`, `accepted`, `rejected`, `completed`, `superseded`, `archived`

**`review_cycle`**: `weekly`, `monthly`, `quarterly`, `yearly`, `never`

**`closure_status`**: `open`, `partial`, `closed`, `archived`

**`decision_status`** (decision notes only): `accepted`, `rejected`, `pending`

## C. MDX artifacts and usage guidance

Every component below is registered globally. No imports needed -- use the JSX tags directly in `.mdx` files. Use these to make content engaging and scannable, but each artifact should earn its place. Never overuse.

### Callout

Coloured boxes for context. Five types:

```
<Callout title="Note" type="info">
  Supplementary context the reader should know.
</Callout>

<Callout title="Warning" type="warn">
  This assumption breaks under specific conditions.
</Callout>

<Callout title="Failure condition" type="error">
  Critical failure if this is wrong.
</Callout>

<Callout title="Verified" type="success">
  This pattern has been validated.
</Callout>

<Callout title="Open question" type="idea">
  An unresolved question worth investigating.
</Callout>
```

| Note type | Where to use |
|-----------|-------------|
| Concept | `warn` in Drawbacks, `info` in Where this fits, `idea` in Open questions |
| Method | `error` in Failure modes, `success` in Validation, `warn` in Drawbacks |
| System | `error` in Failure modes, `warn` in Configuration contract, `info` in Monitoring |
| Decision | `warn` in Tradeoffs accepted, `info` in Context |
| Experiment | `idea` in Interpretation, `error` in Threats to validity |
| Map | `info` in What this map helps answer, `warn` in Gaps and open loops |
| Standard | `error` in Failure conditions, `warn` in Quality bar |

### Steps

Numbered step indicators with vertical progress line:

```
<Steps>
<Step>

### Step title

Step content.

</Step>
<Step>

### Next step

More content.

</Step>
</Steps>
```

| Note type | Where to use |
|-----------|-------------|
| Method | Procedure section (primary use case) |
| Experiment | Setup and Procedure sections |
| System | Architecture section for sequential processes |
| Standard | Required sections when listing order |

### Tabs

Switchable panels for comparisons:

```
<Tabs items={['Option A', 'Option B']}>
<Tab value="Option A">

Content for option A.

</Tab>
<Tab value="Option B">

Content for option B.

</Tab>
</Tabs>
```

Use `groupId` and `persist` to sync tab selection: `<Tabs groupId="lang" items={['Python', 'Bash']} persist>`

| Note type | Where to use |
|-----------|-------------|
| Concept | Comparing approaches (cosine vs dot product) |
| Method | Code sketch in multiple languages |
| System | Config examples for different environments |
| Decision | Options considered (one tab per alternative) |
| Standard | Body template vs sidecar template per type |

### Code blocks

Supports titles, line numbers, highlighting, diffs, and focus:

````
```python title="example.py" lineNumbers
def example():
    return True  # [!code highlight]
```
````

Annotations (place at end of line inside fence):

| Annotation | Effect |
|-----------|--------|
| `// [!code highlight]` | Highlights the line |
| `// [!code ++]` | Addition (green) |
| `// [!code --]` | Removal (red) |
| `// [!code focus]` | Dims other lines |
| `// [!code word:term]` | Highlights occurrences of `term` |

| Note type | Where to use |
|-----------|-------------|
| Method | Code sketch section (primary use) |
| System | Configuration contract, Architecture |
| Concept | Worked example when code clarifies math |
| Experiment | Setup and Procedure |

### KaTeX math

Inline: `$k = 60$` renders as inline math.

Display:

```
$$
\cos(\theta) = \frac{\mathbf{a}\cdot\mathbf{b}}{\|\mathbf{a}\|\|\mathbf{b}\|}
$$
```

| Note type | Where to use |
|-----------|-------------|
| Concept | Equation section, Breakdown (primary use) |
| Method | Formal rule or formula |
| Experiment | Results (quantitative findings) |

### Wiki-links

Use `[[wiki_key]]` to link between notes. Resolved links become clickable; unresolved render as plain text.

```
- Parent map: [[Learning_Map]]
- Related: [[cosine_similarity]], [[dot_product]]
- Used in project: [[ukb_pipeline]]
```

Use wiki-links in every Where this fits, Related notes, Notes in this map, and Where I have used it section.

## D. Closure rules and quality standards

### Closure dimensions (U-L-D-A-E-M)

Every note is scored 0-6 across six binary dimensions:

| Dim | Name | Satisfied when |
|-----|------|---------------|
| U | Upward | Links to at least one parent map |
| L | Lateral | Links to at least two peer notes or prerequisites |
| D | Downward | Links to at least one child, example, or comparison |
| A | Applied | Links to at least one project, system, or experiment |
| E | Epistemic | States limitations, failure modes, or open questions |
| M | Maintenance | Has status, review cycle, and last-reviewed date |

### Closure by note type

**Concept**: parent map (U), two peers/prerequisites (L), one child/example (D), one method/system/project (A), open questions (E), review metadata (M).

**Method**: parent map (U), concept dependencies via `uses_concepts` (L), implementation via `implemented_in` (D+A), validation via `validated_by`, project link (A), open questions (E), review metadata (M).

**System**: parent map (U), method link via `implements_methods`, dependency links via `depends_on` (L), project link via `used_in_projects` (A), open questions (E), review metadata (M).

**Decision**: context, options (`alternatives`), why chosen, tradeoffs, consequences, review trigger, `deciders`, `date_decided`.

**Experiment**: question, hypothesis, setup, procedure, results, interpretation, threats to validity, feedback links (`related_methods`/`related_systems`), `date_started`.

**Project**: `key_methods` (>= 1), `key_systems` (>= 1), `related_notes` (>= 1), open questions (E), review metadata (M).

**Map**: `children` (>= 1), review metadata (M).

### Promotion ladder

| Status | Meaning |
|--------|---------|
| `draft` | Incomplete or partially linked |
| `active` | Useful and mostly linked, still changing |
| `evergreen` | Stable, linked, reviewed, reusable |
| `archived` | Kept for history only |

No note may be promoted to `evergreen` below 5/6 closure.

### The seven writing rules

1. **Define before explaining** -- state what it is in the first sentence
2. **Stable section order** -- never reorder sections within a note type
3. **Plain-English interpretation** -- after every equation or formal rule, explain in plain English
4. **One worked example** -- every concept, method, and system note needs a concrete example
5. **Show drawbacks** -- no note only sells the idea; drawbacks section must not be empty
6. **Link outward** -- at least two `[[wiki_links]]` to other notes
7. **Decision utility** -- reader leaves knowing what to do, when to use it, when not to

### Type-specific additions

**Math/Concept**: equation in `$$...$$`, breakdown with variable definitions, plain-English interpretation, worked example, ASCII diagram.

**Method**: inputs/outputs, procedure as numbered steps, code sketch, validation section.

**System**: architecture diagram, configuration contract, failure modes, monitoring.

**Decision**: options considered (>= 2), why chosen, tradeoffs accepted, review trigger.

**Experiment**: hypothesis before results, setup/procedure separated, threats to validity, feedback to other notes.

### Failure conditions

- `.mdx` starting with `---` (frontmatter) fails the materialiser build
- Misspelled component tags render as plain text or cause build errors
- Missing any of the seven writing rules makes a note incomplete
- Concept without worked example cannot reach closure
- Method without procedure cannot reach closure
- System without failure modes cannot reach closure
- Inline `style={}` props with hardcoded colours are forbidden
