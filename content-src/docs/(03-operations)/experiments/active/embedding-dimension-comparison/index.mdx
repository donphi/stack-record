## Question

Does increasing embedding dimensionality from 768 to 1024 produce a measurable improvement in retrieval precision and recall on the [[ukb_pipeline]] biomedical corpus?

## Hypothesis

A 1024-dimensional biomedical embedding model will improve Precision@10 by at least 5 percentage points over the 768-dimensional baseline on the same test query set, because the additional dimensions encode finer-grained semantic distinctions relevant to biomedical terminology.

## Why this matters

The [[ukb_pipeline]] retrieval stage uses dense embeddings to rank candidate documents before reranking via [[rrf_pipeline]]. Embedding quality directly determines the recall ceiling — documents missed at the retrieval stage cannot be recovered downstream. If the 1024-dim model provides meaningfully better recall, the increased GPU memory cost may be justified. If the gains are marginal, the 768-dim model should be retained to keep inference costs lower and batch sizes larger.

## Setup

### Models under test

| Model | Dimensions | Parameters | Source |
|---|---|---|---|
| PubMedBERT-base | 768 | 110M | Hugging Face `microsoft/BiomedNLP-PubMedBERT-base-uncased-abstract` |
| BioLORD-2023-L | 1024 | 335M | Hugging Face `FremyCompany/BioLORD-2023-L` |

### Corpus

- 50,000 PubMed abstracts sampled from the UK Biobank phenotype-linked subset.
- Pre-tokenised and stored in Parquet format.

### Query set

- 200 manually curated queries spanning 10 biomedical domains (cardiovascular, neurological, metabolic, etc.).
- Each query has 20 relevance-judged documents (binary relevant/not-relevant labels).

### Infrastructure

- Embedding generation runs on the [[gpu_worker_pool]] (2× A100 40GB).
- Vector indices built with FAISS `IVF4096,PQ64` for the 768-dim model and `IVF4096,PQ128` for the 1024-dim model.
- Retrieval latency measured as p50 and p99 over the full query set.

## Procedure

1. Encode all 50,000 abstracts with both models. Store embeddings in separate FAISS indices.
2. Run all 200 queries against each index. Retrieve top-50 candidates per query.
3. Compute Precision@10, Recall@50, and MRR for each model.
4. Compare memory footprint: peak GPU memory during encoding and index size on disk.
5. Run a paired bootstrap significance test (10,000 iterations) on Precision@10 differences.
6. Record retrieval latency (p50, p99) for both indices under identical hardware.

## Results

In progress — awaiting benchmark completion. Encoding for the 1024-dim model is 60% complete as of 2026-04-16. Preliminary numbers on the first 100 queries suggest a 2–3 point Precision@10 improvement, but the full query set is needed before drawing conclusions.

## Interpretation

In progress — awaiting benchmark completion. Early observations:

- The 1024-dim model appears to separate near-synonym biomedical terms (e.g., "myocardial infarction" vs. "heart attack") more cleanly in embedding space.
- Encoding throughput is approximately 40% slower due to larger model size.

Final interpretation will address whether the precision delta justifies the memory and latency costs within the [[ukb_pipeline]] budget.

## Threats to validity

- **Query set bias** — the 200 queries were curated by a single annotator. Domain-specific blind spots may skew results toward or against either model.
- **PQ compression artefacts** — product quantisation may disproportionately harm the 1024-dim model if the PQ codebook size is not proportionally scaled.
- **Corpus size** — 50,000 abstracts is a subset. Retrieval dynamics may differ at the full corpus scale (2M+ abstracts).
- **Single hardware configuration** — latency measurements on A100s may not generalise to smaller GPUs used in production.

## What this changes

If the 1024-dim model wins decisively:
- Update the [[ukb_pipeline]] embedding stage to use BioLORD-2023-L.
- Resize the FAISS index and adjust [[gpu_worker_pool]] memory allocation.
- Re-benchmark the [[rrf_pipeline]] end-to-end to confirm downstream gains.

If the 768-dim model is retained:
- Document the ceiling and close this experiment.
- Consider alternative retrieval improvements (query expansion, hybrid sparse-dense).
